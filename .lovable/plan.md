

# Secure GitHub Publishing

## Problem
The `.env` file contains backend credentials (project ID, anon key, URL) and is not in `.gitignore`. Additionally, `DevAutoLogin.tsx` has hardcoded fallback credentials. Several `.md` files expose the project ID in URLs.

## Important Context
The `VITE_SUPABASE_PUBLISHABLE_KEY` (anon key) and `VITE_SUPABASE_URL` are **public by design** -- they are embedded in the client-side JavaScript bundle that every visitor downloads. They are NOT secrets. Security comes from RLS policies, not from hiding these values. However, it's still best practice to load them from environment variables rather than committing them to Git.

The `.env` file is auto-generated by Lovable Cloud and should not be in the repo. The real secrets (service role key, API keys) are already safely stored as edge function secrets and never appear in code.

## Plan

### 1. Update `.gitignore`
Add these entries:
```
.env
.env.*
.env.local
```

### 2. Create `.env.example`
A template file (safe to commit) showing required variables without values:
```
VITE_SUPABASE_PROJECT_ID=""
VITE_SUPABASE_PUBLISHABLE_KEY=""
VITE_SUPABASE_URL=""
VITE_DEV_EMAIL=""
VITE_DEV_PASSWORD=""
```

### 3. Remove hardcoded dev credentials from `DevAutoLogin.tsx`
Change the fallback values to empty strings so no credentials leak:
```typescript
const DEV_EMAIL = import.meta.env.VITE_DEV_EMAIL || "";
const DEV_PASSWORD = import.meta.env.VITE_DEV_PASSWORD || "";
```
Add an early return if either is empty (show "configure env vars" message).

### 4. Redact project IDs from documentation files
In `PRE_LAUNCH_CHECKLIST.md`, `TIKTOK_SANDBOX_SETUP.md`, `EXTERNAL_SETUP_REQUIRED.md`, and `CHATGPT_SECURITY_REVIEW_PROMPT.md` -- replace hardcoded `fmvbzhlqzzwzciqgbzgp` URLs with placeholder like `YOUR_PROJECT_REF`.

### 5. Email templates (edge functions) -- NO change needed
The Supabase project URL in edge function email templates runs server-side only and is already available via `Deno.env.get('SUPABASE_URL')`. However, for the logo URLs these are public storage URLs that need to be hardcoded (they're public bucket assets, not secrets). These are fine to keep.
